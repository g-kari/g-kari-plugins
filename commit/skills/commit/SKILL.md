# Commit

git commit のワークフローを自動化するスキル。

## ワークフロー

### Step 1: 変更の確認

以下のコマンドを**並列で**実行し、現在の変更状態を把握する：

```bash
git status            # untracked / modified / staged の確認
git diff              # unstaged の差分
git diff --staged     # staged の差分
git log --oneline -5  # 直近のコミットメッセージスタイル確認
```

**判断ポイント:**
- 変更がゼロならユーザーに報告して終了
- untracked ファイルに `.env` / `credentials` / 秘密情報が含まれていないかチェック

### Step 2: プロジェクトのチェックコマンド検出と実行

コミット前に実行すべきチェックを**プロジェクトの設定ファイルから自動検出**する：

| 検出先 | 探すキー | 実行コマンド例 |
|---|---|---|
| `package.json` scripts | `check-all`, `check`, `lint`, `type-check` | `npm run check-all` |
| `Makefile` | `check`, `lint`, `test` | `make check` |
| `.pre-commit-config.yaml` | （存在すれば） | `pre-commit run --all-files` |
| `CLAUDE.md` / `.claude/rules/git.md` | コミット前の指示 | 記載されたコマンド |

**検出の優先順位:**
1. `CLAUDE.md` や `.claude/rules/` にコミット前の指示があればそれに従う
2. `package.json` に `check-all` があればそれを実行
3. 個別の `lint` / `type-check` / `format:check` があればそれぞれ実行
4. 何も見つからなければチェックをスキップ

チェックが失敗した場合：
- エラー内容をユーザーに報告
- 自動修正可能なら修正を提案（format の場合は `npm run format` 等）
- 修正後に再度チェックを実行

### Step 3: ステージングの確認

以下のルールでステージングを判断する：

| 状態 | アクション |
|---|---|
| staged な変更がある | そのまま使用 |
| staged が空で unstaged がある | 関連するファイルを `git add` で個別にステージ |
| untracked ファイルがある | 内容を確認し、コミットに含めるべきか判断 |

**重要:**
- `git add -A` や `git add .` は使わない。ファイルを個別に指定する
- `.env` / `credentials.json` / シークレット系は絶対にステージしない
- 機能的に無関係な変更が混在している場合、分割コミットを提案する

### Step 4: コミットメッセージの生成

#### 引数でメッセージが指定された場合（`/commit -m 'メッセージ'`）
そのメッセージをそのまま使用する。

#### メッセージが指定されていない場合
差分を分析してコミットメッセージを生成する：

1. **直近のコミットログからスタイルを推定する**
   - 日本語 or 英語
   - prefix の有無（`feat:` / `fix:` 等）
   - 1行 or 複数行

2. **差分の性質を分類する**

| 分類 | 判断基準 |
|---|---|
| 新機能 | 新ファイル追加、新しい関数/クラス/エンドポイント |
| 改善 | 既存機能への変更・拡張 |
| バグ修正 | エラーハンドリング、条件分岐の修正 |
| リファクタリング | 動作を変えない構造変更 |
| テスト | テストファイルの追加・修正 |
| ドキュメント | README, CLAUDE.md, コメントの変更 |
| 設定 | config, CI/CD, 依存関係の変更 |

3. **メッセージを生成する**
   - 「何を変えたか」ではなく「なぜ変えたか」にフォーカス
   - 1〜2文で簡潔に
   - プロジェクトの既存スタイルに合わせる

### Step 5: コミットの実行

```bash
git commit -m "$(cat <<'EOF'
<生成したコミットメッセージ>
EOF
)"
```

コミットが成功したら `git status` で結果を確認し、ユーザーに報告する。

**pre-commit hook でコミットが失敗した場合:**
- エラー内容を確認して原因を修正
- **`--amend` は絶対に使わない** — 新しいコミットを作成する
- `--no-verify` は使わない — hook の失敗は修正すべき問題

## 引数

| 引数 | 説明 | 例 |
|---|---|---|
| `-m 'メッセージ'` | コミットメッセージを直接指定 | `/commit -m '初期実装'` |
| `--push` | コミット後に push も実行 | `/commit --push` |
| （なし） | 差分から自動生成 | `/commit` |

## 注意事項

- シークレットファイルは絶対にコミットしない
- `--no-verify` / `--no-gpg-sign` は使わない
- `git add -A` / `git add .` は使わない
- `--amend` は使わない（前のコミットを破壊するリスクがある）
- force push はしない
- push は `--push` 引数がある場合のみ実行する
